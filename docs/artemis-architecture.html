Manifest Structure capability declaration
Every module implements get_manifest() — this is the complete interface contract
# ModuleManifest — complete capability declaration name: str # unique snake_case identifier version: str # semver icon: str # Material icon name color: str # hex — used for UI theming entities: List[EntitySchema] # data models this module owns widgets: List[WidgetDefinition] # Flutter widget descriptors dashboard_cards: List[DashboardCardDefinition] actions: List[ActionDefinition] schedulable_activities: List[ActivityDefinition] # calendar-compatible tasks notification_types: List[NotificationDefinition] provides_data: List[DataProviderDefinition] # data shared with other modules consumes_data: List[DataConsumerDefinition] # data needed from other modules metrics: List[MetricDefinition] # trackable analytics requires_modules: List[str] # dependency declarations has_settings: bool settings_schema: Optional[Dict]

Cross-Module Integration Points permissioned data sharing
All sharing is declared in manifests and requires user-granted permissions stored in cross_module_shares
Provider Module Data Exposed Consumer Module(s) Use Case Permission
Work Calendar availability, work hours Fitness Nutrition Schedule workouts & meals around work blocks read
Fitness Calories burned, activity level, intensity Nutrition Adjust daily calorie targets based on exercise read
Nutrition Calorie intake, macro targets Fitness Fuel workout recommendations read
Sleep Sleep quality score, wake & sleep times Work Fitness Adjust task load & workout intensity from rest quality read
Mindfulness Stress level, mood score Work Fitness Recommend recovery days, reduce load on high-stress days read
Finance Budget categories, spending data Nutrition Meal plan within grocery budget read
Learning Study sessions, focus blocks Work Reserve focus time for deep work and study read
Modules cannot read each other's data directly. All cross-module data access goes through the Core Platform's permissioned sharing layer, with records stored in cross_module_shares. Users grant or revoke permissions at any time.
Integration Patterns how sharing works
Provider declaration — in manifest.provides_data
id: "calories_burned" name: "Daily Calories Burned" description: "Per-day calorie burn + activity type" schema: {date, calories, activity_type} query_action: "get_daily_calories"
Consumer declaration — in manifest.consumes_data
id: "fitness_calories" provider_module: "fitness" provider_id: "calories_burned" required: false use_case: "Adjust calorie targets from exercise"

Database Schema postgresql 15+
Key tables — all use UUID primary keys and JSONB for flexible payload fields
TableKey FieldsPurposeIndexes
module_registry name PK, version, manifest JSONB, enabled Active module catalog name
scheduled_activities id UUID, module_name FK, activity_type, scheduled_time, status, data JSONB Activity orchestration state scheduled_time, status, module_name
notifications id UUID, module_name FK, type, priority, read, actions JSONB, related_activity_id FK Unified notification store read, created_at
user_dashboards user_id + card_id PK, module_name, widget_id, position JSONB, config JSONB Per-user dashboard config user_id
feedback id UUID, module_name, activity_id FK, rating SMALLINT, notes, data JSONB Activity feedback & insights module_name, activity_id
cross_module_shares provider, consumer, provider_id — UNIQUE constraint; permission, expires_at Permissioned data sharing provider+consumer
